# 외부망의 경우
hansulee-home.conf 에 다음을 추가
```
# Fix the routing
route vpn_gateway 255.255.255.255
route 10.0.0.0 255.192.0.0 vpn_gateway

# Fix the packet size
tun-mtu 1420
```

위의 명령어는 아래의 명령어와 같음:
```
# 게이트웨어 주소를 tun0 device로 던지기
sudo ip route add 10.8.0.1 dev tun0 
# 10.0.0.0/10 대역의 주소는 모두 10.8.0.1 -> device tun0으로 던지기
sudo ip route add 10.0.0.0/10 via 10.8.0.1 dev tun0
# 둘 다 필요함
```


```
sudo systemctl restart openvpn@hansulee-home
```

restart 후에 routing table이 업데이트 된 것을 확인
```
ip route
default via 192.168.0.1 dev wlp128s20f3 proto dhcp metric 600 
10.0.0.0/10 via 10.8.0.1 dev tun0

10.8.0.1 via 10.8.0.1 dev tun0

172.17.0.0/16 dev docker0 proto kernel scope link src 172.17.0.1 linkdown 
192.168.0.0/24 dev wlp128s20f3 proto kernel scope link src 192.168.0.21 metric 600 
```

```conf
# prevention of all traffics destined to 10.x.x.x to be routed via tun interface
route 10.6.0.0 255.255.0.0 net_gateway
route 221.158.185.6 255.255.255.255 net_gateway
# force traffics destined to 10.10.0.14 to tun interface
route 10.10.0.14 255.255.255.255 


# 3. MTU 최적화 (끊김 방지)
tun-mtu 1420
mssfix 1380
```

# 최종 문제 해결을 위해 시도한 부분 (제미나이)
오, 아주 날카로운 지적입니다! 네트워크 설계를 직접 하셔도 될 정도로 논리적이에요. 말씀하신 방식이 바로 **'스플릿 터널링(Split Tunneling)'**의 핵심 개념입니다.

질문하신 내용들 하나씩 짚어보겠습니다.

---

### 1. "10.6.x.x만 제외하고 나머지를 VPN으로 보내면 안 되나?"

네, 이론적으로 아주 훌륭한 방법입니다. 하지만 실제 구현에서는 약간의 우선순위 문제가 있습니다.

- **가장 구체적인 지도가 이긴다:** 네트워크는 "범위가 좁고 구체적인 길"을 우선순위로 둡니다. 현재 사용자님의 로컬 망은 `10.6.0.0/16`이고, VPN이 주려는 사내망은 `10.0.0.0/10`입니다.
    
- **리눅스의 거부 이유:** 사용자님이 말씀하신 대로 `10.6.x.x`를 보호하려고 해도, VPN 서버가 던져준 `10.0.0.0/10`이라는 거대한 범위가 사용자님의 "진짜 대문(`10.6.0.1`)"까지 삼켜버리려고 하니까 리눅스가 **"이건 자살행위야!"**라며 에러를 내고 길을 안 만드는 겁니다.
    

**결론적으로:** 사용자님의 아이디어를 OpenVPN 설정으로 번역하면 **"이 범위(10.6.x.x)는 절대 VPN을 타지 말고 원래 랜선(`net_gateway`)으로 가라"**고 명시하는 것이 됩니다.

---

### 2. `ip route`에서 `tun`이 안 보이는 이유

**"프로세스가 죽었기 때문"입니다.**

마지막 로그 끝에 `^C`와 `SIGINT... process exiting` 보이죠? 사용자님이 터미널에서 강제로 종료했거나 에러로 인해 OpenVPN이 꺼진 상태입니다.

- OpenVPN은 꺼지는 순간 자신이 만들었던 가상 랜카드(`tun`)와 그와 관련된 모든 길(Route)을 흔적도 없이 지워버립니다.
    
- 그래서 지금 보시는 `ip route` 결과는 **VPN이 연결되지 않은 순수 생태계 상태**인 것이 맞습니다. 제대로 바뀐 게 아니라, **연결에 실패해서 원상복구 된 것**이죠.
    

---

### 3. 해결책: 사용자님의 아이디어를 반영한 최종 설정

사용자님의 제안("내 로컬 망은 건드리지 마!")을 반영해서 설정을 다듬어 보겠습니다. `/etc/openvpn/hansulee-home.conf` 맨 밑에 아래 내용을 넣어주세요.

Plaintext

```
# 1. 내 로컬 망(10.6.x.x)은 무조건 원래 랜선으로 가라 (사용자님의 아이디어)
route 10.6.0.0 255.255.0.0 net_gateway

# 2. VPN 서버 접속 주소도 원래 랜선으로 가라 (연결 유지 필수)
route 221.158.185.6 255.255.255.255 net_gateway

# 3. MTU 최적화 (끊김 방지)
tun-mtu 1420
mssfix 1380

# 4. 라우팅 적용 전 딜레이 (충돌 방지)
route-delay 5
```

### 4. 실행 및 확인 순서

1. **설정 저장** 후 서비스를 다시 시작합니다.
    
    Bash
    
    ```
    sudo systemctl restart openvpn@hansulee-home
    ```
    
2. **5~10초 대기** (route-delay 때문에 시간이 필요합니다).
    
3. **상태 확인**
    
    Bash
    
    ```
    ip route
    ```
    
    이때 **`tun0` 혹은 `tun1`이 포함된 줄이 보여야 성공**입니다.
    
4. **핑 테스트**
    
    Bash
    
    ```
    ping 10.10.0.14
    ```
    

**자, 이렇게 설정을 바꾸고 다시 시작했을 때 `ip route`에 드디어 `tun`이 등장하나요?** 만약 등장한다면 `10.10.0.14`가 잡히는지 확인해 주세요! (안 보인다면 `sudo journalctl -u openvpn@hansulee-home -f`로 실시간 로그를 다시 한번 봐야 합니다.)

# 🧭 OpenVPN 라우팅 문제 전체 정리 (재현 가능 문서)

---

# 0️⃣ 최초 문제 상황

### 증상

VPN은 연결된 것 같은데, 내부 Git 서버 접속이 안 됨.

```bash
ping 10.10.0.14
```

출력:

```
From 10.6.0.1 icmp_seq=1 Destination Net Prohibited
```

---

# 1️⃣ 1차 관찰: 이게 무슨 의미인가?

### 해석

- `10.10.0.14` 로 가는 패킷이
    
- VPN이 아니라
    
- 기본 게이트웨이 `10.6.0.1` 로 나가고 있음
    
- 그리고 거기서 "Destination Net Prohibited" 로 차단당함
    

즉:

> ❗ VPN을 타지 않고 일반 사내망 라우터로 나가고 있다.

---

# 2️⃣ VPN이 진짜 연결된 건지 확인

```bash
sudo systemctl status openvpn@hansulee-home
```

핵심 출력:

```
Initialization Sequence Completed
TUN/TAP device tun0 opened
net_addr_v4_add: 10.8.250.74/16 dev tun0
```

### 해석

- TLS 인증 성공
    
- 서버 인증서 검증 OK
    
- tun0 생성됨
    
- 10.8.250.74 IP 부여됨
    

✔ 결론:

> VPN 터널 생성과 인증은 정상.

문제는 "연결"이 아니라 **라우팅**임.

---

# 3️⃣ 서버가 어떤 라우트를 밀어넣었는지 확인

OpenVPN 로그:

```
PUSH_REPLY,
route-gateway 10.8.0.1,
route 10.8.0.0 255.255.0.0,
route 10.0.0.0 255.192.0.0,
```

### 이 의미는?

서버가 클라이언트에게 이렇게 말한 것:

- 10.8.0.0/16 → VPN으로 보내
    
- 10.0.0.0/10 → VPN으로 보내
    

여기서 핵심:

```
10.0.0.0/10
```

이 범위에는 **10.6.x.x도 포함됨**

---

# 4️⃣ 문제의 본질 발견

현재 PC 실제 네트워크:

```
10.6.250.9
게이트웨이: 10.6.0.1
```

즉:

- 내가 이미 10.6.x.x 대역 안에 있음
    
- 그런데 VPN 서버가 10.0.0.0/10 전체를 VPN으로 보내라고 함
    
- 이 안에 10.6.x.x도 포함됨
    

그러면 무슨 일이 생기냐?

### 💥 논리 충돌 발생

만약 10.0.0.0/10을 VPN으로 보내버리면:

- 내 현재 네트워크 (10.6.x.x)
    
- 기본 게이트웨이 (10.6.0.1)
    
- VPN 서버 통신 경로
    

모두 VPN으로 보내게 됨

→ VPN 터널 유지에 필요한 트래픽까지 터널 안으로 들어감  
→ 서버와 통신 끊김  
→ ping-restart 발생

실제 로그:

```
Inactivity timeout (--ping-restart), restarting
```

✔ 이걸로 확정:

> VPN이 자기 자신을 가둬버리는 구조적 충돌

---

# 5️⃣ 라우팅 테이블 확인

```bash
ip route
```

출력:

```
default via 10.6.0.1 dev enp12s0
10.6.0.0/16 dev enp12s0
```

tun 인터페이스 관련 경로 없음.

✔ 해석:

- 서버가 route push 했지만
    
- 리눅스 커널이 충돌로 거부했거나
    
- 연결이 끊기면서 사라짐


성공 (에러 없음)

```bash

```

```
10.10.0.14 dev tun0
```

그리고:

```bash
ping 10.10.0.14
```

응답 성공.

✔ 결론:

> 터널은 정상이고, 문제는 자동 라우팅 충돌이 맞다.

---

# 7️⃣ 근본 해결 전략

해결 목표:

1. 내 로컬망(10.6.x.x)은 VPN 안 타게
    
2. VPN 서버 주소는 절대 VPN으로 보내지 않게
    
3. MTU 불일치 해결
    

---

# 8️⃣ 최종 적용한 설정

`/etc/openvpn/hansulee-home.conf`에 추가:

```
route 10.6.0.0 255.255.0.0 net_gateway
route 221.158.185.6 255.255.255.255 net_gateway
tun-mtu 1420
mssfix 1380
route-delay 5
```

---

# 9️⃣ 왜 이 설정이 해결이 되었는가?

### 1️⃣ net_gateway의 의미

VPN 연결 직전 사용 중이던 기본 게이트웨이로 보내라는 뜻.

즉:

- 사내망이면 → 10.6.0.1
    
- 집이면 → 192.168.x.1
    
- 카페면 → 그 카페 공유기
    

동적으로 변함.

---

### 2️⃣ 10.6.0.0 예외 처리

10.6.x.x는 절대 VPN으로 보내지 않도록 강제.

1. VPN 연결 성공 ≠ 라우팅 성공

## 🛠 OpenVPN 연결 문제 해결 리포트

### 1. 문제 발견 (Symptoms)

- **상황:** VPN은 연결된 것 같은데, 특정 사내 서버(`git.rtst.co.kr` / `10.10.0.14`) 접속이 안 됨.
    
- **확인 커맨드 & 아웃풋:**
    
    Bash
    
    ```
    ping 10.10.0.14
    # 결과: From 10.6.0.1 icmp_seq=1 Destination Net Prohibited
    ```
    
    - **판단:** 내 컴퓨터가 데이터를 VPN(`tun0`)이 아닌, 일반 사내망 대문(`10.6.0.1`)으로 던지고 있음. 즉, **라우팅(길찾기) 실패.**
        

---

### 2. 원인 분석 (Diagnosis)

- **로그 확인 커맨드:** `sudo systemctl status openvpn@hansulee-home` 또는 `journalctl`
    
- **발견된 핵심 에러:**
    
    1. `ERROR: Linux route add command failed`: 서버가 준 길(Route)을 내 PC가 거부함.
        
    2. `Inactivity timeout (--ping-restart)`: 연결이 7초마다 끊기고 재시작됨 (무한 루프).
        
- **문제 원인 파악:**
    
    - **네트워크 충돌:** VPN 서버가 "10.0.0.0 대역은 다 VPN으로 가!"라고 명령했는데, 이 범위에 현재 내가 쓰고 있는 실제 IP 대역(`10.6.x.x`)이 포함됨.
        
    - **리눅스의 방어:** "지금 쓰고 있는 랜선 주소를 VPN으로 돌리면 인터넷 끊겨!"라며 리눅스 커널이 이 명령을 차단함.
        
    - **자기 발등 찍기:** VPN 서버 주소로 가는 길마저 VPN 터널 안으로 숨어버려서 서버와 통신이 끊기고 재시작되는 현상 발생.
        

---

### 3. 해결 방법 (Solution)

`/etc/openvpn/hansulee-home.conf` 파일 맨 밑에 아래 설정을 추가하여 **'내 랜선 주소'**와 **'VPN 서버 주소'**를 보호함.

**수정된 설정값:**

Plaintext

```
# 1. 내 로컬망(10.6.x.x)은 VPN을 타지 말고 원래 랜선(net_gateway)으로 가라
route 10.6.0.0 255.255.0.0 net_gateway

# 2. VPN 서버 실제 IP 주소도 원래 랜선으로 가야 연결이 안 끊김
route 221.158.185.6 255.255.255.255 net_gateway

# 3. 데이터가 너무 커서 끊기지 않도록 크기 조절 (MTU 최적화)
tun-mtu 1420
mssfix 1380

# 4. VPN 연결 직후 길을 찾을 시간을 벌어줌
route-delay 5
```

---

### 4. 최종 확인 (Verification)

설정 적용 후 아래 순서로 확인하여 성공을 확정함.

1. **서비스 재시작:** `sudo systemctl restart openvpn@hansulee-home`
    
2. **라우팅 테이블 확인:**
    
    Bash
    
    ```
    ip route
    # 결과에 '10.10.0.14 dev tun0' 또는 
    # '10.0.0.0/10 via 10.8.0.1 dev tun0' 형식이 있으면 성공
    ```
    
3. **최종 핑 테스트:**
    
    Bash
    
    ```
    ping 10.10.0.14  # 이제 응답이 옴!
    ```
